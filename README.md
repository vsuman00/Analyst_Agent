# Analyst Agent

**Analyst Agent** is an enterprise-grade backend system that converts GitHub repositories into comprehensive Business Requirement Documents (BRDs). The pipeline is **deterministic-first** — all structural decisions are made by rule-based agents. **OpenAI** is used selectively in an optional enrichment layer to improve the professional quality of feature descriptions and the executive summary without altering any structural data.

---

## System Architecture

The pipeline uses a multi-layered, state-machine driven architecture to ensure high-fidelity context extraction and absolute data provenance.

**Pipeline Flow:**
```
Client → FastAPI → Pipeline Controller → ECA Engine → Context Builder
       → Normalizer → Validator → Analysis Agents → Requirement Generators
       → [LLM Enrichment Layer] → BRD Composer → Validator → Export (.md / .docx)
```

- **Layer 1: Architecture (`architecture/`)** — Technical SOPs in Markdown.
- **Layer 2: Navigation** — Routes data between SOPs and tools.
- **Layer 3: Tools (`app/`)** — Deterministic Python scripts. Atomic and testable.

---

## Pipeline State Machine

```
CREATED → INGESTING → ECA_DONE → CONTEXT_READY → NORMALIZED → VALIDATED
        → ANALYZED → [LLM_ENRICHED] → BRD_READY → VALIDATING → IMPROVING
        → COMPLETED / FAILED
```

## How to Setup and Run

### 1. Prerequisites

- Python 3.9+
- An OpenAI API key (for LLM enrichment — the pipeline runs in deterministic-only mode without it)

### 2. Installation

```bash
# Create a virtual environment
python -m venv .venv
```

**Activate the virtual environment:**

- **macOS / Linux:**
  ```bash
  source .venv/bin/activate
  ```

- **Windows:**
  ```powershell
  .\.venv\Scripts\activate
  ```

**Install dependencies:**

```bash
pip install -r requirements.txt
```

### 3. Environment Configuration

```bash
# Copy the example env file
cp .env.example .env
```

Open `.env` and set your OpenAI API key:

```env
OPENAI_API_KEY=sk-your-key-here
OPENAI_MODEL=gpt-4o-mini         # Optional, default: gpt-4o-mini
OPENAI_MAX_TOKENS=2048           # Optional, default: 2048
```

> **Note:** If `OPENAI_API_KEY` is not set, the pipeline will run in **deterministic-only mode** — all BRD content will be generated by rule-based agents. LLM enrichment is silently skipped.

### 4. Running the Application

```bash
# Ensure your virtual environment is activated, then run:
python -m uvicorn app.api.main:app --reload
```

The API is available at `http://localhost:8000`.  
Interactive Swagger docs: `http://localhost:8000/docs`

## LLM Integration

The OpenAI API is used in **Phase 3.5 — LLM Enrichment**, which runs between the analysis agents and the BRD composer. It enriches three specific outputs:

| Enrichment Target | LLM Task | Fallback |
|---|---|---|
| **Feature Descriptions** | Rewrites terse extracted descriptions into precise SHALL-style sentences | Original deterministic description |
| **Executive Summary** | Generates a ≤120-word summary grounded in the pipeline context | Template-based deterministic summary |
| **Core Value Statement** | Writes a ≤30-word value delivery statement from the feature list | Concatenated feature name list |

All prompts supply the full structured context. `temperature=0` is enforced. If any LLM call fails, the deterministic fallback is used — **the pipeline never blocks on LLM availability**.

## Pipeline Stages

### Stage 1: ECA (Extract, Classify, Aggregate)
- Clone repository
- Build file tree (ignoring `node_modules`, `build`, binaries)
- Classify files (`src/`, `routes/`, `services/`, `README`)
- Extract and chunk content

### Stage 2: Context Intelligence
- Aggregate data
- Normalize features
- Validate completeness
- Apply token budgets and control limits

### Stage 3: Rule-Based Analysis
- **FeatureExtractionAgent** — Scans codebase against a keyword registry.
- **FeatureValidator** — Merges, deduplicates, and scores features.
- **FeatureInterpretationAgent** — Converts signals into interpreted features with evidence.
- **BusinessUnderstandingAgent** — Derives `product_type`, `primary_users`, `core_value`.
- **ProductUnderstandingAgent** — Derives product archetype from feature clusters.
- **FunctionalRequirementGenerator** — Maps features to 1–3 testable FRs with acceptance criteria.
- **NonFunctionalRequirementGenerator** — Generates 6–8 system-level NFRs from tech stack signals.

### Stage 3.5: LLM Enrichment _(requires OPENAI_API_KEY)_
- Enrich feature descriptions via OpenAI (JSON mode, temperature=0)
- Generate executive summary
- Refine core value statement

### Stage 4: BRD Composition
- **BRDComposer** — Assembles all structured inputs into an 8-section enterprise BRD.
- **BRDValidator** — Scores completeness, traceability, no-hallucination, clarity (threshold: 0.85).
- **FixLoop** — Applies up to 2 deterministic repair passes if score < 0.85.

### Stage 5: Export
- **DocumentGenerator** — Converts final BRD Markdown to professional `.docx`.

## Error & Failure Handling

- **No API Key**: Pipeline runs fully deterministically. LLM step is silently skipped.
- **LLM API Error**: Individual enrichment calls fall back to their deterministic outputs. Pipeline never blocks.
- **Missing README in Repo**: System dynamically infers purpose from code structure.
- **Self-Annealing Loop**: BRDValidator scores the BRD → FixLoop applies targeted repairs (max 2 iterations).

---

## Design & Behavioral Principles

1. **Reliability First**: Deterministic pipeline always produces a valid BRD. LLM is optional enrichment.
2. **Data-First**: JSON Schemas are defined before code. All outputs are Pydantic-validated.
3. **No Hallucination**: LLM prompts supply full structured context. `temperature=0`. Outputs validated.
4. **Absolute Traceability**: Every feature maps to evidence. Every FR maps to a feature. BRDValidator enforces this.
